## [2026-04-09 14:55] Задача: План создания аддона "Energy Management System (EMS)"

### Archi
- **Архитектура**: Полноценный Docker-аддон для Home Assistant. Backend на Python (FastAPI/SQLAlchemy) для логики и интеграции с HA API.
- **Интерфейс**: Веб-панель (Ingress), где пользователь выбирает сенсоры из выпадающего списка (через WebSocket API HA) и задает параметры динамических нагрузок.
- **Алгоритм**: 
    - Сбор статистики за N дней для создания "тепловой карты" потребления дома.
    - Матрица принятия решений: Сравнение прогноза солнечной генерации, профиля потребления и цен на энергию.
    - Динамические нагрузки: Очередь запуска (бойлер -> стиралка) с учетом приоритетов и цен.
- **Масштабируемость**: Возможность легкого добавления новых типов инверторов и батарей через абстрактные классы.

### Skeptic
1. **Безопасность и Отказоустойчивость**: Аддон управляет реальным силовым оборудованием. Ошибка в логике может перегреть бойлер или глубоко разрядить батарею. Нужны жесткие "физические" лимиты (Hard Limits), которые нельзя обойти через UI.
2. **Проблема данных**: Пользователь может ошибиться при выборе сенсора (перепутать Вт и кВт). Система должна валидировать входящие данные и иметь режим "симуляции" перед запуском реального управления.
3. **Ресурсы (SD-карты)**: Постоянная запись профилей потребления в базу данных на Raspberry Pi быстро изнашивает память. Нужно использовать SQLite с редким сбросом на диск (WAL mode) или хранить краткосрочные данные в памяти.

### Заключение
Принято решение строить систему по модульному принципу:
- Модуль **Monitor** (только сбор данных и валидация).
- Модуль **Statistic** (расчет профилей).
- Модуль **Brain** (математическая оптимизация).
- Модуль **Actuator** (безопасное управление нагрузками).
Первым делом создаем структуру аддона и механизм выбора сенсоров.

## [2026-04-09 15:08] Задача: Уточнение требований (нагрузки, прогноз, люди)

### Archi
- **Классификация нагрузок**: Разделяем на Циклические (стиралка) и Нециклические (бойлер). Бойлер выносим в отдельный приоритетный модуль с учетом температуры.
- **Прогнозирование**: Добавляем модуль сравнения "Прогноз vs Реальность". Это позволит системе динамически корректировать стратегию, если генерация ниже ожидаемой.
- **Человеческий фактор**: Вводим коэффициент "Количество людей". Это позволит динамически менять "безопасный остаток" (Reserve) в батарее на вечер.
- **История**: Выносим глубину анализа (N дней) в настройки пользователя.

### Skeptic
1. **Сложность Бойлера**: Бойлеры инертны. Если просто включать/выключать его по цене, можно оставить людей без горячей воды. Нужна интеграция с целевой температурой и расчет времени нагрева.
2. **Проблема Прогноза**: Если мы видим, что прогноз "врет", система не должна впадать в панику. Нужен алгоритм сглаживания (фильтр Калмана или экспоненциальное сглаживание) для коррекции планов.
3. **Зависимость от Людей**: Сенсоры присутствия часто ошибаются. Если в доме 0 людей по датчику, а по факту они есть, система может выключить бойлер в неподходящий момент. Нужен "ручной переключатель" или учет времени суток.

### Заключение
- Бойлер выделяется в отдельный конфигурационный блок.
- Вводим понятие `BaseReserve` + `OccupancyCorrection`.
- Добавляем логику обучения на ошибках прогноза.
- Реализуем поддержку сенсоров "Активного цикла" для умных розеток.

## [2026-04-09 15:10] Задача: Финализация стратегии (Solcast, Температура, Приоритеты)

### Archi
- **Solcast**: Интегрируем API Solcast для получения почасовых прогнозов. Реализуем сравнение "Прогноз vs Факт" для корректировки планов.
- **Boiler Logic**: Используем сенсор температуры для расчета теплоемкости. Логика: "Греем до целевой, когда есть лишнее солнце или цена < X, но не в ущерб Survival SOC".
- **Survival SOC**: Динамический расчет минимального заряда батареи, необходимого для питания дома до рассвета (с учетом текущего прогноза потребления и кол-ва людей).

### Skeptic
1. **Лимиты Solcast**: Бесплатный API имеет лимиты. Нужно кешировать данные и не запрашивать их слишком часто.
2. **Безопасность Бойлера**: Программный контроль температуры должен иметь гистерезис, чтобы не дергать реле.
3. **Критический Приоритет**: Если батарея ниже "Уровня выживания", все нагрузки (включая бойлер) должны быть безусловно отключены, пока заряд не восстановится.

### Заключение
- Стратегия "Батарея важнее Бойлера" принята за основу.
- Используем Solcast (через сенсоры Home Assistant) как основной источник данных о солнце.
- Прямое обращение к внешним API не требуется.
- Приступаем к созданию проекта.

## [2026-04-09 15:13] Задача: Принципы системной инженерии (DRY, Модульность)

### Archi
- **Модульность**: Весь код разбивается на независимые сервисы. Математика (статистика, профилирование) отделена от логики интеграции с HA.
- **DRY**: Любая логика, используемая более одного раза (например, нормализация данных сенсоров или логирование), выносится в `utils` и `base_classes`.
- **Интерфейсы**: Используем типизацию (Type Hints) и абстрактные классы для легкой поддержки и расширения системы.

### Skeptic

2. **Тестируемость**: Модульный подход обязывает нас писать тесты для каждого сервиса в отдельности.
3. **Производительность**: Многослойная архитектура на Python должна быть оптимизирована, чтобы не создавать задержек в цикле управления.

### Заключение
- Принята архитектура Service-Provider.
- Все общие функции выносятся в ядро (`core/`).
- Реализуем строгую структуру папок для разделения ответственности.

## [2026-04-09 15:15] Задача: Дополнительные системные идеи (Persistence, Dry Run, Notifications)

### Archi
- **Persistence (Состояние)**: Сохраняем стейт активных циклов в SQLite. Если аддон перезагрузится, он должен "вспомнить", что стиралка в процессе, а не запускать ее заново.
- **Dry Run (Режим аудита)**: Глобальный переключатель, при котором `Actuator` только пишет логи "Я бы включил бойлер", но не шлет команды в HA. Позволяет пользователю проверить логику.
- **Умные уведомления**: Интеграция с `notify` в HA. Система сообщает о важных действиях (например: "Начата продажа энергии по пиковой цене").
- **Watchdog**: Если аддон теряет связь с HA или самодиагностика провалена, все нагрузки переводятся в "безопасное состояние" (например, выключаются).

### Skeptic
1. **Сложность Dry Run**: Нужно гарантировать на уровне кода, что в режиме аудита физические команды *физически* не могут уйти.
2. **Заспамленность**: Уведомления должны быть настраиваемыми по уровням (INFO, WARNING, CRITICAL).
3. **Проблема перезагрузки**: Если аддон "вспомнит" цикл после долгого простоя (напр. питание дома пропало на час), цикл может быть уже неактуален. Нужен таймаут актуальности стейта.

### Заключение
- Добавляем `DryRun` как базовую настройку.
- Реализуем `StateService` для хранения текущих процессов.
- Планируем модульную систему уведомлений.

## [2026-04-09 15:23] Задача: Определение состояний инвертора (FSM)

### Archi
- **Конечный автомат**: Реализуем логику через State Machine с 7 состояниями (`buy`, `bat_emergency`, `stop_sale`, `no_pv_sale_no_bat`, `sale_pv`, `sale_pv_no_bat`, `sale_pv_bat`).
- **Прозрачность**: Текущее состояние инвертора будет отображаться в UI с пояснением, *почему* оно выбрано (например: "Режим stop_sale из-за низкого SOC").

### Skeptic
1. **Защита от "дребезга" (Flapping)**: Переключения между состояниями не должны происходить слишком часто (нужен минимальный таймаут пребывания в состоянии).
2. **Конфликты**: Нужно четко определить приоритеты. Если одновременно "Низкая цена" (хочется `buy`) и "Аварийный SOC" (хочется `bat_emergency`) — что выигрывает? (В данном случае `buy`, так как он восстановит SOC быстрее).
3. **Безопасность**: Переход в `sale_pv_bat` должен быть защищен двойной проверкой цены и прогноза на завтра.

### Заключение
- Схема из 7 состояний утверждена.
- Реализуем через Python Enum и сервис `InverterController`.
- Добавляем гистерезис и задержки на переключение для защиты оборудования.

## [2026-04-09 15:55] Задача: Модульность нагрузок (Plugin Architecture)
