## [2026-04-09 14:55] Задача: План создания аддона "Energy Management System (EMS)"

### Archi
- **Архитектура**: Полноценный Docker-аддон для Home Assistant. Backend на Python (FastAPI/SQLAlchemy) для логики и интеграции с HA API.
- **Интерфейс**: Веб-панель (Ingress), где пользователь выбирает сенсоры из выпадающего списка (через WebSocket API HA) и задает параметры динамических нагрузок.
- **Алгоритм**: 
    - Сбор статистики за N дней для создания "тепловой карты" потребления дома.
    - Матрица принятия решений: Сравнение прогноза солнечной генерации, профиля потребления и цен на энергию.
    - Динамические нагрузки: Очередь запуска (бойлер -> стиралка) с учетом приоритетов и цен.
- **Масштабируемость**: Возможность легкого добавления новых типов инверторов и батарей через абстрактные классы.

### Skeptic
1. **Безопасность и Отказоустойчивость**: Аддон управляет реальным силовым оборудованием. Ошибка в логике может перегреть бойлер или глубоко разрядить батарею. Нужны жесткие "физические" лимиты (Hard Limits), которые нельзя обойти через UI.
2. **Проблема данных**: Пользователь может ошибиться при выборе сенсора (перепутать Вт и кВт). Система должна валидировать входящие данные и иметь режим "симуляции" перед запуском реального управления.
3. **Ресурсы (SD-карты)**: Постоянная запись профилей потребления в базу данных на Raspberry Pi быстро изнашивает память. Нужно использовать SQLite с редким сбросом на диск (WAL mode) или хранить краткосрочные данные в памяти.

### Заключение
Принято решение строить систему по модульному принципу:
- Модуль **Monitor** (только сбор данных и валидация).
- Модуль **Statistic** (расчет профилей).
- Модуль **Brain** (математическая оптимизация).
- Модуль **Actuator** (безопасное управление нагрузками).
Первым делом создаем структуру аддона и механизм выбора сенсоров.

3. **Производительность**: Многослойная архитектура на Python должна быть оптимизирована, чтобы не создавать задержек в цикле управления.

## [2026-04-09 15:15] Задача: Дополнительные системные идеи (Persistence, Dry Run, Notifications)

### Archi
- **Persistence (Состояние)**: Сохраняем стейт активных циклов в SQLite. Если аддон перезагрузится, он должен "вспомнить", что стиралка в процессе, а не запускать ее заново.
- **Dry Run (Режим аудита)**: Глобальный переключатель, при котором `Actuator` только пишет логи "Я бы включил бойлер", но не шлет команды в HA. Позволяет пользователю проверить логику.
- **Умные уведомления**: Интеграция с `notify` в HA. Система сообщает о важных действиях (например: "Начата продажа энергии по пиковой цене").
- **Watchdog**: Если аддон теряет связь с HA или самодиагностика провалена, все нагрузки переводятся в "безопасное состояние" (например, выключаются).

### Skeptic
1. **Сложность Dry Run**: Нужно гарантировать на уровне кода, что в режиме аудита физические команды *физически* не могут уйти.
2. **Заспамленность**: Уведомления должны быть настраиваемыми по уровням (INFO, WARNING, CRITICAL).
3. **Проблема перезагрузки**: Если аддон "вспомнит" цикл после долгого простоя (напр. питание дома пропало на час), цикл может быть уже неактуален. Нужен таймаут актуальности стейта.

### Заключение
- Добавляем `DryRun` как базовую настройку.
- Реализуем `StateService` для хранения текущих процессов.
- Планируем модульную систему уведомлений.

## v1.3.23: Hourly House Consumption History Implementation

**Archi**: "To build a truly smart EMS, we need to know the house consumption profiles. I propose mirroring the `SolarHourlyStat` pattern to store hourly kWh for the home. This will allow us to predict nightly battery drain and adjust the 'Survival SOC' dynamically. I'll also add an 'N Weeks' retention setting in the UI."

**Skeptic**: 
1. **DB Growth**: Every new hourly table adds weight. We must enforce strict pruning based on the `N * 7` rule to keep the SQLite file manageable.
2. **Sensor Resets**: Unlike solar (which usually only goes up), home energy sensors can reset or behave weirdly if the inverter is rebooted. We need a 'safe delta' check: if `current < start`, we either use the raw current or skip the delta to avoid negative/mammoth values.
3. **Redundancy**: Why not just use the existing `SensorHistory` table? 
   * *Archi Response*: `SensorHistory` is for raw time-series. `HourlyStat` is summarized, making forecasting calculations MUCH faster.

**Consolidated Plan**: 
- Implement `HouseHourlyStat` as a separate table for performance. 
- Implement robust delta calculation in `main.py` with reset protection.
- Add a configurable `history_weeks` setting with a default of 4 weeks.
- Strict pruning in the hourly loop.

**Skeptic Approval**: ✅ Одобрено. (Consolidated as safe for deployment).
